# Make Codex ticket drafting asynchronous with a pending ticket state

## Context

Currently the ticket creation flow waits for Codex drafting to complete before the user can continue. Change the flow so submitting "Draft with Codex" immediately closes the creation modal, creates a visible pending/drafting ticket in Todo, and lets the Codex draft continue in the background. The user should be able to click back into that ticket while drafting is still running, then see the finished draft when the run completes.

## Review Feedback

- The create-ticket modal must close instantly after the user starts a Codex draft.
- A ticket must appear immediately in the Todo column.
- The Todo-column ticket must show an active spinner icon or equivalent active drafting indicator while Codex is drafting.
- If the user opens the ticket before drafting finishes, the ticket detail view must show a loading state until the drafted ticket content is ready.

## Research Findings

- tests/ticket-draft.test.ts covers the existing draft path with createTicketDraft, draftToCreateInput, extractTicketDraftUrls, TicketDraftCodexClient, TicketDraftDependencies, TicketDraftThread, readyStatus, createProject, and projectPath; this is the likely test surface for async draft behavior.
- src/main/services/codex/research.ts defines TicketDraftResearchContext, CandidateResearchFile, DEFAULT_TICKET_DRAFT_RESEARCH_LIMITS, and related create-draft research plumbing, so background drafting must preserve the bounded research context behavior.
- src/main/services/codex/index.ts contains Codex SDK integration types such as Thread, ThreadEvent, ThreadItem, ThreadOptions, AgentTicketUpdate, and AgentTicketUpdateInput; this is the likely service layer for keeping a Codex drafting run alive after the modal closes.
- src/main/services/schemas.ts defines ticket draft and run-related schemas including CreateDraftInput, EpicSubticketCreateInput, RelayCodexEvent, ticketPrioritySchema, ticketTypeSchema, and runStatusSchema; update or reuse these schemas for a pending/drafting ticket lifecycle if needed.
- src/shared/types.ts defines board statuses RELAY_TODO_STATUS, RELAY_IN_PROGRESS_STATUS, RELAY_NEEDS_CLARIFICATION_STATUS, RELAY_REVIEW_STATUS, RELAY_NOT_DOING_STATUS, and RELAY_COMPLETED_STATUS. There is no existing board column named Pending, so this should likely be modeled as ticket draft/run state while the ticket remains in Todo unless product direction says otherwise.
- Research was limited to 160 scanned files and 6 read files; UI component locations for the create-ticket modal were not identified in the supplied research context.

## Requirements

- When the user submits "Draft with Codex", the UI must not block until Codex completes drafting.
- The create-ticket modal/dialog must close immediately after the draft request is accepted locally.
- A new ticket must be created immediately in Todo with a clear pending/drafting state and enough information for the user to recognize it from the rough idea they entered.
- The pending Todo ticket must display an active spinner icon or equivalent animated active indicator while the Codex draft run is in progress.
- The background Codex draft run must continue after the modal closes and must update the same ticket when complete.
- Clicking the pending ticket before completion must open the ticket detail view and show a loading state while drafting is still in progress.
- The detail loading state must be replaced by the drafted ticket content when Codex finishes successfully.
- When drafting succeeds, the ticket must be populated with the generated title, priority, labels, context, research findings, requirements, implementation plan, acceptance criteria, clarification questions, implementation notes, ticket type, and subtickets as applicable.
- When drafting fails, the pending ticket must remain accessible and show a recoverable error state rather than disappearing silently.
- Users must be able to start multiple Codex ticket drafts in quick succession without waiting for previous drafts to complete.
- Existing synchronous draft tests should be updated or extended to cover the asynchronous pending-ticket lifecycle.

## Implementation Plan

- Locate the create-ticket UI flow that calls createTicketDraft and identify where it currently awaits completion before closing the modal.
- Introduce or reuse a ticket-level draft/run status such as drafting, draft_failed, and draft_complete that is independent from board status; keep newly created pending draft tickets in RELAY_TODO_STATUS unless a product decision adds a new board column.
- Change the "Draft with Codex" submit handler to create a placeholder ticket immediately from the user's rough idea, persist its pending/drafting metadata, close the modal, and then start the Codex draft run in the background.
- Thread the placeholder ticket id through the Codex draft service so completion and failure events update the existing ticket instead of creating a separate ticket after the draft is ready.
- Ensure the background run uses the existing bounded research path in src/main/services/codex/research.ts and preserves current schema validation from src/main/services/schemas.ts.
- Update Todo ticket rendering to show an active spinner icon while the ticket is in the drafting state.
- Update ticket detail rendering so drafting tickets show a loading state instead of incomplete draft content until the Codex run completes.
- Update ticket detail/list rendering to show pending drafting state, completion state, and failure state for tickets whose Codex draft has not finished.
- Add concurrency handling so multiple pending draft tickets can exist at once and each Codex run updates only its associated ticket.
- Extend tests in tests/ticket-draft.test.ts or nearby UI/service tests to assert immediate placeholder creation, non-blocking return, spinner/active indicator state, detail loading state, eventual draft population, failure persistence, and multiple simultaneous drafts.
- Run the relevant unit tests and any available typecheck/lint commands for the touched modules.

## Acceptance Criteria

- Submitting "Draft with Codex" returns control to the user and closes the modal before the Codex run completes.
- A pending/drafting ticket appears in Todo immediately after submission.
- The pending Todo ticket displays an active spinner icon or equivalent active drafting indicator while Codex is drafting.
- The pending ticket can be opened while drafting is still running and displays a loading state until the ticket draft is ready.
- Starting several drafts back to back creates several distinct pending tickets without waiting on prior Codex runs.
- When a Codex run completes successfully, the corresponding pending ticket is updated with the generated draft content and no duplicate ticket is created.
- If a Codex run fails, the corresponding ticket remains visible with an error state and the user's original idea is preserved.
- Automated tests cover the asynchronous lifecycle, including the Todo spinner/active indicator and ticket-detail loading state, and pass locally.

## Clarification Questions

- Should "pending state" be a ticket-level drafting indicator within Todo, or should Relay add a separate Pending board column?

## Implementation Notes

- Do not block the UI on Codex completion; the only synchronous requirement is durable creation of the placeholder ticket and background-run association.
- The Todo card needs a visibly active spinner icon or equivalent animated status indicator while `runStatus` is drafting.
- The ticket detail view should avoid showing partial or empty generated fields as if they were final content; show a clear loading state until drafting succeeds or fails.
- Be careful to distinguish board status from Codex draft/run status because src/shared/types.ts does not currently define a Pending board status.
- If a migration or schema change is needed for persisted draft/run state, include backwards-compatible handling for existing tickets.
- The supplied research did not identify the exact frontend modal component, so implementation should begin by tracing createTicketDraft usage from the UI call site.
